## Cross-Cutting Findings

These apply to **all 5 SKILL.md** — fixing once standardizes the whole pack.

1. **No SSL frontmatter at all.** Every skill has `name`/`version`/`description` but no formal `scheduling`/`structural`/`logical` blocks. Triggers live inside `description:` as prose. anti-triggers, side-effects, idempotency, rollback are nowhere declared in machine-readable form.

3. **`side_effects` not declared in frontmatter** — even though every skill writes to `.harnish/assets/*.jsonl` and most write `docs/*.md`. Currently a reader must scan the body to see what gets written.
4. **`idempotent` flag missing everywhere.** All 5 are non-idempotent (asset jsonl appends, PRD overwrites, progress mutations) but none say so.
5. **`rollback` missing or implicit.** Only ralphi declares `Test FAIL → rollback that fix` — and that's in body, not frontmatter.

## v1 claims that were wrong or unverified

**E1. ✗ "asset dedup 없음 → re-runs produce duplicate rows"**
(claimed in `drafti-architect` Top Risk #3 and `drafti-feature` Top Risk #2)

Wrong. [scripts/harnish_py/record.py:76-84](scripts/harnish_py/record.py:76) implements slug dedup — same title appends `-2`, `-3` suffix to keep slugs unique. Content dedup is a different question (still absent — same payload with different titles produces two rows), but the unqualified "duplicate rows" claim is false. Withdraw and replace with: *"`record-asset.sh` dedups on slug only, not content. Re-running the same skill on the same problem with identical title produces a single row; with rewritten title produces two."*

**E2. ⚠ "purge-assets.sh is not user-reachable from any skill"**
(claimed in v1 Tool Dependency Graph footnote)

Half-wrong. Re-checked: no utterance trigger in any SKILL.md maps to purge — confirmed by `grep -E "(purge|폐기)" skills/*/SKILL.md` returning nothing. **However**, [skills/impl/references/retention-policy.md:23](skills/impl/references/retention-policy.md:23) documents the call flow including step 4: *"원본에서 candidates 제거"* — i.e. the deletion is a real operation, just invoked outside the skill surface (CLI / cron / pre-commit hook). Withdraw the "unreachable" framing and replace with: *"purge-assets.sh has destructive semantics (asset removal from `harnish-assets.jsonl`) but no skill utterance reaches it; invocation is out-of-band."*

**E3. ⚠ Scripts treated as opaque — but they're 5-line Python wrappers**

All scripts I cited as "tools used by impl" — `record-asset.sh`, `query-assets.sh`, `validate-progress.sh`, `loop-step.sh`, `compress-progress.sh`, `compress-assets.sh`, `check-thresholds.sh`, `check-violations.sh`, `skillify.sh`, `quality-gate.sh` — are all **5-line wrappers** that `exec python3 -m harnish_py <cmd>`. Real logic lives in [scripts/harnish_py/](scripts/harnish_py/) (~20 modules). v1 evaluated SSL Logical based on shell call sites only.

**Implication for the audit framework itself**: the proposed `tools:` frontmatter field is insufficient when wrappers hide the real implementation. To make SSL Logical declarations verifiable, the field should resolve to actual implementation paths — e.g. `tools: ["scripts/harnish_py/record.py:record-asset"]`, not `tools: ["scripts/record-asset.sh"]`.

**E4. ✗ Score numerics (75, 88, 52, 38, ...) are heuristic, not systematic**

The auditor spec says "누락 1건당 -15점, 모호 1건당 -8점, Logical ×1.5". I did not systematically count violations — the numbers are eyeballed. Treat the ranking (which skill is riskier than which) as informative, the absolute numbers as decorative. **A real auditor should print the violation list with deductions per item, not just a final score.**

**E5. ⚠ "drafti" trigger conflict was overclaimed**

v1 framed bare `"drafti"` matching both drafti-architect and drafti-feature as a hard conflict. Both descriptions actually carry prose disambiguation: *"(when no planning doc is provided)"* and *"or when a planning document is attached/referenced"*. A router that reads `description:` fully — which Claude Code does for skill matching — has the cue. The conflict only bites a router doing keyword-only match against the trigger list. Re-frame: *"frontmatter is router-agnostic, but the disambiguation cue lives in prose; a structured `anti_triggers:` would make it explicit."*

## New findings v1 missed

**N1. orphan reference in impl — same pattern as ralphi**

`skills/impl/references/retention-policy.md` exists but is **never loaded** by `impl/SKILL.md` body. Confirmed by `grep -rn "retention-policy" skills/*/SKILL.md` → no hits. The Step 1 Mode Detection table loads `thresholds.md` for asset-related utterances, never `retention-policy.md`. This is the same orphan pattern as ralphi's `criteria-project.md`. Different severity though: ralphi's orphan corresponds to a missing inspection mode, impl's corresponds to a deletion policy that exists outside any skill flow.

**N2. deletion side-effect at the system level is unbound to any skill's SSL surface**

`purge-assets.sh` (via `harnish_py/purge.py`) deletes records from `harnish-assets.jsonl`. The flow is documented in `retention-policy.md` (an unloaded impl reference, see N1). The result: the harnish ecosystem **has** a destructive op, but no SKILL.md frontmatter declares `side_effects.deletes` for it. Whether this is a bug depends on intent — if purge is purposefully out-of-band (CLI/cron only), the SSL frame should still surface it as "system-level side effect, not skill-invoked"; if it's meant to be skill-reachable, both N1 and N2 are real omissions.

**N3. Wrapper-vs-implementation split is itself an SSL audit blind spot**

See E3. Generalizing: any audit that reads SKILL.md alone will under-report side effects when scripts delegate to a separate runtime (Python module, MCP server, sibling CLI). The auditor design needs a "follow-through" step — for each tool referenced in SKILL.md, resolve to its actual implementation file and re-audit side effects there. This applies to the auditor spec itself.
