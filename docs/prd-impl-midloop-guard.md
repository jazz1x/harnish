## 1. Problem Definition

### 1.1 As-Is
harnish:impl v0.0.2 사용 중, 7/20 태스크 완료 시점에 블로커(PRD 충돌/파괴 작업/하드 가드) 없이 "Due to the size and complexity, let me create a quick summary" 형태로 mid-phase에서 자발적으로 중단하는 사례 발생. `SKILL.md:320-328` Termination Conditions는 `Todo empty` / `user says stop` / `session end` 세 가지만 정의하며, milestone HITL(phase 경계, L176)이 허용 중단점. 그 외 "컨텍스트 피로/요약 도피"를 금지하는 명시 조항 없음 → 회색지대.

### 1.2 To-Be
Step 3 루프 중 중단 가능한 시점이 **milestone HITL / escalation(3 failures) / hard-guardrail 위반** 세 가지뿐임이 문서상 명시. SKILL.md/ko.md에 grep 가능한 금지 문구가 존재하며, bats 스위트가 해당 문구 존재를 회귀 검증.

### 1.3 Impact
autonomous 계약이 스킬 정체성(설명문 첫 줄 "Autonomous implementation engine"). 계약 위반 사례가 재현 가능한 상태로 릴리스되어 있음 → 패치 없이는 다음 사용자에게도 동일 재현.

## 2. Design Decision

### Alternative A: 명시적 금지 조항 (Negative constraint, prose-only)
- **Approach**: Step 3에 "Prohibited mid-loop" 절 추가 + Hard Guardrails에 대칭 한 줄. `scripts/` 무변경, 런타임 훅 없음.
- **Pros**: 기존 가드와 강제 레벨 동형(prose). 플러그인 버전 정렬(README/plugin.json/모든 SKILL frontmatter/VERSION/CHANGELOG)은 어차피 릴리스 오버헤드로 발생하므로 순수 변경 표면은 impl SKILL 본문 2곳.
- **Cons**: 런타임 강제력 없음 — LLM이 문서를 따르지 않으면 동일 재발 가능. bats는 존재만 검증.
- **Suitable**: 스킬 자체가 prose-driven이고, 다른 가드도 모두 문서 기반일 때.

### Alternative B: 허용 중단점 화이트리스트 중심 재구성
- **Approach**: Termination Conditions를 "유일한 허용 중단점" 섹션으로 재정의하고 "이 외 모든 중단 금지" 선언. Step 3 내 mid-loop 언급은 해당 섹션 참조로 짧게 처리.
- **Pros**: 단일 진실 소스(SSOT). 향후 허용 지점이 늘거나 줄 때 한 곳만 수정.
- **Cons**: Termination Conditions 섹션명/구조 변경이라 기존 mental model 파괴. milestone HITL(Step 3 내부)과 Termination Conditions(별도 섹션)가 레벨이 다름 → 통합 시 어색.
- **Suitable**: 스킬을 1.0 메이저 리팩터 할 때.

### Alternative C: 런타임 훅 (scripts/loop-step.sh 출력에 리마인더)
- **Approach**: loop-step.sh가 매 호출 시 "Only stop on: milestone HITL / 3-failure escalation / hard-guardrail" 문자열을 stdout에 찍어 LLM 컨텍스트에 주입.
- **Pros**: 루프 진입마다 reminder, 문서보다 강제력 높음.
- **Cons**: prose-driven 스킬 철학과 결 다름. 다른 가드와 비대칭. 테스트 표면 증가.
- **Suitable**: 재발 시 A만으로 부족하다고 판명된 다음 단계.

### Selection: **A**
- **Rationale**: 패치 성격의 v0.0.3 bump. 기존 스킬 모든 가드가 prose인데 이것만 런타임 강제는 비대칭. 재발 여부는 A 릴리스 후 관찰 → 필요 시 C로 escalate.
- **Rejected**: B는 오버엔지니어링(구조 리팩터). C는 현시점 과함.
- **Validity condition**: A 릴리스 후에도 동일 mid-loop 중단이 재현되면 C(런타임 훅)로 재검토.

### 4.2 추가할 문구 (en)

Step 3 끝에 새 섹션:

```markdown
### Prohibited Mid-Loop

The only permitted stop points inside Step 3 are:
1. **Milestone HITL** — at phase boundary (see "On Phase Completion")
2. **Escalation** — 3 consecutive acceptance_criteria failures
3. **Hard guardrail violation** — see Guardrails

Any other interruption is prohibited — including but not limited to:
- Asking the user for permission to continue without a blocker
- Proposing a mid-phase summary/checkpoint due to context size or task count
- Stopping because "many tasks completed" without a defined phase boundary

Context fatigue is not a stop condition. If context is tight, rely on `compress-progress.sh` and continue.
```

Hard Guardrails에 추가 한 줄:

```markdown
- Mid-loop interruption outside milestone HITL / 3-failure escalation / hard-guardrail violation prohibited
```

### 4.2-ko 추가할 문구 (ko) — 리터럴 고정

Step 3 끝에 새 섹션:

```markdown
### 루프 중단 금지

Step 3 내부에서 허용되는 중단 지점은 다음 셋뿐입니다:
1. **Milestone HITL** — phase 경계 ("On Phase Completion" 참고)
2. **Escalation** — acceptance_criteria 3회 연속 실패
3. **Hard guardrail 위반** — Guardrails 참고

그 외 모든 중단은 금지 — 다음을 포함하되 이에 한정되지 않음:
- 블로커 없이 "계속 진행해도 되는지" 사용자에게 묻기
- 컨텍스트 크기나 태스크 수를 이유로 mid-phase 요약/체크포인트 제안
- 정의된 phase 경계 없이 "태스크를 많이 완료했으니" 중단

컨텍스트 피로는 중단 사유가 아닙니다. 컨텍스트가 빠듯하면 `compress-progress.sh`를 쓰고 계속 진행합니다.
```

Hard Guardrails에 추가 한 줄:

```markdown
- milestone HITL / 3회 실패 escalation / hard-guardrail 위반 외의 mid-loop 중단 금지
```

## 7. Guardrails

### Soft
- Step 3 섹션 구조(READ/ACT/LOG/PROGRESS)는 건드리지 않음 — 금지 절은 맨 끝에 부착
- Hard Guardrails는 기존 4개 유지, 신규 1줄만 append

### Hard
- Termination Conditions 섹션의 기존 3개 조건 수정/삭제 금지 (허용 중단점 목록은 여기 아님, Step 3 내부)
- `harnish-current-work.json` 스키마 변경 금지
- `scripts/` 아래 변경 금지 (A는 문서-only 접근)
